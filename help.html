This system uses a Node.js backend server connected to a MySQL database running inside XAMPP.

Prerequisites

  • XAMPP installed with MySQL module
  • Node.js v16 or later

Step 1 — Start XAMPP MySQL

  1. Open the XAMPP Control Panel.
  2. Click Start next to MySQL (and optionally Apache).

Step 2 — Create the Database

  1. Open phpMyAdmin in your browser.
  2. Click the Import tab at the top.
  3. Choose the file database.sql from this project folder.
  4. Click Go. This creates the cida_machinery database and its tables.

Alternatively, run via command line:

mysql -u root -p < database.sql

Step 3 — Install Node Dependencies

Open a terminal in the project folder and run:

npm install

Step 4 — Start the Server

node server.js

The server starts at http://localhost:8000. Open that URL in your browser.

MySQL Connection Settings

The server connects using XAMPP default settings:

  • Host: localhost
  • User: root
  • Password: (empty)
  • Database: cida_machinery

If you have set a MySQL root password in XAMPP, open server.js and update the password field in the pool configuration.

Contractor Feature Flow

  1. A contractor registers at contractor-register.html. Their account is saved with status pending.
  2. A CIDA admin logs in and navigates to Contractor Management in the admin dashboard to approve or reject the account.
  3. Once approved, the contractor can log in at contractor-login.html.
  4. After login, contractors can browse available machinery and submit rental requests from the contractor dashboard.

The CIDA Machinery Registration System is a browser-based application built for Sri Lanka's Construction Industry Development Authority. It manages the registration, certification, renewal, and maintenance tracking of construction machinery used across the country.

Purpose

  • Provide a centralised digital register of approved construction machinery.
  • Allow machinery owners to submit registration applications with supporting documents.
  • Enable the Director General to review, certify, reject, or revoke registrations.
  • Let the maintenance admin track equipment servicing, repairs, and schedules.

User Roles

  1. Public User — View the approved register (no login required).
  2. Owner — Register machinery, pay fees, view status, renew, and appeal.
  3. Maintenance Admin — Create and manage maintenance records, view alerts and reports.
  4. Director General — Certify or reject applications, manage the approved register, review appeals, export reports.
  • HTML5 — Semantic markup with data-page routing attributes.
  • Vanilla JavaScript (IIFE / ES5) — No frameworks; all logic wrapped in IIFEs and exposed via global objects.
  • CSS3 — Custom properties (variables), grid/flexbox layout, responsive media queries.
  • localStorage — Persistent data store for users, machinery, and maintenance tables.
  • sessionStorage — Session management (current user token).
  • Browser APIs — Blob (CSV/PDF export), FormData (form handling), window.open (print certificates).

All data is stored in localStorage as JSON arrays. The CIDA_DB global object provides a simple CRUD API that every page module uses.

CIDA_DB API (db.js) 
CIDA_DB.getData(table)          // Returns an array of records
CIDA_DB.saveData(table, data)   // Overwrites the table with data
CIDA_DB.insert(table, record)   // Appends a record with generated ID
CIDA_DB.update(table, id, fields) // Merges fields into matching record
CIDA_DB.findById(table, id)     // Returns a single record or null

ID Generation

IDs follow the pattern {table_prefix}_{timestamp}_{random}, e.g. m_1717012345678_4821 for a machinery record. The prefix is the first character of the table name.

Tables

  • users — All user accounts (owners, admin, director general).
  • machinery — Registration records submitted by owners.
  • maintenance — Equipment maintenance entries managed by the admin.

Migration & Seeding

migration.js runs on every page load. It checks whether seed data exists and, if not, inserts default user accounts (admin, director general, owner) and sample machinery records so the system works immediately on first visit.

Global Objects

  • CIDA_DB — Data access layer (CRUD operations on localStorage).
  • CIDA_AUTH — Authentication helpers (getSession, setSession, clearSession, requireRole, redirectForRole).
  • CIDA_UTILS — Shared utilities (escapeHtml, formatDate, addYears, setFeedback, getText, status helpers).
  • CIDA_CONSTANTS — Static reference data (machineryTypes, documentTypes).

Script Loading Order

Every page loads scripts in dependency order: db.js → migration.js → auth.js (if authenticated) → page-specific module. This ensures globals are available before any page logic runs.

Page Routing

Each HTML page declares data-page on the <body> tag (e.g. data-page="owner-dashboard"). After login, CIDA_AUTH.redirectForRole() sends the user to the correct dashboard based on their role.

Event-Driven Re-Rendering

Page modules call render functions after data mutations (insert, update). For example, after a machinery submission the owner page re-reads the table and rebuilds the status board <tbody>.

Modal Pattern

Modals use a <div class="modal" hidden> wrapper with a backdrop and panel. Opening removes the hidden attribute and adds body.modal-open; closing restores both.

5.1 Public User

  • Visit index.html — no login required.
  • View the Approved Register table showing all certified machinery.
  • Use the filter form to search by owner name, machinery type, location, or registration number.
  • Navigate to Register to create an owner account, or Login to access a dashboard.

5.2 Owner

  • Register Machinery — Fill out the registration form with machinery type, make/model, country of origin, and location. Attach 4 required documents: Revenue License, Motor Traffic Certificate, Affidavit, and Engineer Report.
  • Fee Tiers — The registration fee depends on the number of previously registered machines:
    • 1st machine: Rs. 1,000
    • 2nd machine: Rs. 750
    • 3rd machine: Rs. 500
    • 4th machine: Rs. 300
    • 5th+ machines: Rs. 100
  • Status Board — View all submitted machinery with current status (Pending, Approved, Rejected, Revoked, Pending Renewal).
  • Renewal — When a certificate is within 30 days of expiry, a “Renew” button appears. Renewal resets the status to pending_renewal for DG review.
  • View / Print Certificate — For approved machinery, click “View Certificate” to see the digital certificate card and print or save as PDF.
  • Submit Appeal — If a registration is rejected, the owner can submit an appeal with a written explanation for the Director General to review.

5.3 Maintenance Admin

  • Dashboard — View total records, completed count, and scheduled/overdue count. See service vs. repair breakdown and location coverage.
  • Create Record — Fill out the maintenance form with equipment ID, name, date, status, type (service/repair), location, site, and attach documents.
  • History Table — Browse all maintenance records with status badges and document links.
  • Alerts & Notifications — Bell icon shows upcoming scheduled maintenance and overdue items within the configured notification window.
  • Settings — Configure the notification window (days) and default status for new records.

5.4 Director General

  • Review Queue — Table of pending and pending-renewal applications. Each row has “Certify” and “Reject” buttons.
  • Certify — Approving generates a registration number in the format CIDA-{TYPE_CODE}-{YEAR}-{SEQ}, e.g. CIDA-WL-2026-001. Sets expiry to 1 year from approval date.
  • Reject — Prompts for a rejection reason that the owner can see. Status changes to rejected.
  • Approved Register — Searchable table of all certified machinery. The DG can revoke any active certificate.
  • Appeals Review — Table of submitted appeals. The DG can accept (returns machinery to pending for re-review) or dismiss the appeal.
  • Export — Generate CSV or PDF reports of the full registration summary including category breakdown.

Session Management

User sessions are stored in sessionStorage as a JSON object containing userId and role. Sessions are cleared on tab close or explicit logout.

Role-Based Access Control (RBAC)

Protected pages call CIDA_AUTH.requireRole(role) on load. If the session is missing or the role does not match, the user is redirected to login.html.

XSS Prevention

All user-supplied text rendered into the DOM passes through CIDA_UTILS.escapeHtml(), which replaces &, <, >, ", and ' with their HTML entity equivalents.

Prototype Limitations

This is a student prototype and has the following known limitations:

  • Plaintext Passwords — Passwords are stored as-is in localStorage. A production system would use server-side hashing (bcrypt, argon2).
  • No Server — All logic runs in the browser. There is no backend API, database server, or network requests.
  • Simulated File Uploads — Document file inputs capture the file name but do not actually upload or store file contents.
  • No HTTPS / CORS — Since everything runs locally, there are no transport security considerations.
